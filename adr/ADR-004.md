# Título
Adopción de estrategia de ramas develop – qa – main.

# Contexto
El proyecto requiere un flujo organizado de desarrollo para permitir:
- desarrollo paralelo de funcionalidades
- pruebas controladas
- despliegues estables

El equipo de desarrollo utiliza Git como sistema de control de versiones.

# Problema
Trabajar directamente sobre una única rama puede generar:
- conflictos frecuentes
- despliegues inestables
- dificultad para probar cambios antes de producción
  
# Decisión 
Se adopta la siguiente estrategia de ramas:
### main
contiene versiones estables listas para producción.

### qa
rama destinada a pruebas de calidad antes de pasar a producción.

### develop
rama principal de desarrollo donde se integran nuevas funcionalidades.

Flujo de trabajo:
feature → develop → qa → main

# Justificación técnica
Esta estrategia permite:
- aislar el desarrollo de nuevas funcionalidades
- realizar pruebas antes de producción
- mantener una rama estable en todo momento

También facilita procesos de integración continua.

# Consecuencias o impacto esperado
### Positivo
- mayor estabilidad del sistema
- mejor control del ciclo de desarrollo
- pruebas estructuradas antes de producción

### Negativo
- mayor disciplina en la gestión de merges
- necesidad de definir políticas de pull requests
